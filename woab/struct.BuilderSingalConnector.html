Fluent interface for configuring the signal routing from GTK to Actix.

Typically created from BuilderSignal::connector, BuilderSingalConnector can be configured via fluent methods like tag and inhibit and then either:

  • Get passed to connect_signals that connects signals emitted from a freshly created GTK builder to a freshly created Actix actor.
  • Get converted to a SignalRouter via route_to to connect GTK signals - from builders or otherwise - to an Actix actor.

Register a function for setting the return value of the signal.

GTK requires some signals to return a boolean value - true to "inhibit" and not let the signal pass up the inheritance to other handlers, and false to let it. This can be statically configured in the BuilderSignal derive macro with #[signal(inhibit = true) or #[signal(inhibit = false), but sometimes you need to decide based on the signal's parameters. In such cases you can use this function to register a closure. If the closure returns Some, the signal handler will return the value returned by the closure. If it returns None the signal handler will return the value configured in the macro.

enum MySignal {
    ButtonPress(gtk::Button, #[signal(event)] gdk::EventButton), // needs to return a boolean
    ButtonClick, // needs to return nothing
}

fn some_function_that_creates_stuff(factory: &Factories) {
    factory.window.instantiate().actor()
        .connect_signals(MySignal::connector().inhibit(|signal| match signal {
            MySignal::ButtonPress(_, event) => Some(gtk::Inhibit(event.get_button() == 3)), // inhibit on right click
            _ => None, // all other signals should use the default (static) setting
        }))
        .start(MyActor);
}

Create a SignalRouter that routes signals to the actor owning the supplied context.

This method only sets the target of the signal routing - the signal router object provides methods for connecting the signals from GTK. If they are not used no actual signals will be routed.
